주제 #: 1

한 회사는 Amazon EBS 스토리지를 갖춘 단일 m4.2xlarge Amazon EC2 인스턴스에서 레거시 시스템을 실행합니다. EC2 인스턴스는 웹 서버와 자체 관리형 Oracle 데이터베이스를 모두 실행합니다. 12시간마다 EBS 볼륨의 스냅샷이 생성되고 완전히 구성된 EC2 인스턴스에서 AMI가 생성되었습니다.
EC2 인스턴스를 종료한 최근 이벤트로 인해 몇 시간의 가동 중지 시간이 발생했습니다. 애플리케이션은 AMI에서 성공적으로 시작되었지만
EBS 스냅샷의 노후화와 데이터베이스 복구로 인해 8시간의 데이터가 손실되었습니다. 시스템 운영자가 이러한 프로세스를 수동으로 수행하는 동안 시스템도 4시간 동안 다운되었습니다.
가동 중지 시간을 최소화하고 데이터 손실 가능성을 줄이는 아키텍처 변경 사항은 무엇입니까?

  • A. 인스턴스를 자동으로 복구하려면 Amazon CloudWatch 경보를 생성하십시오. 재부팅 시 데이터베이스를 확인하고 복구하는 스크립트를 만듭니다. CloudWatch 경보로 생성된 Amazon SNS 메시지를 운영 팀에 구독하세요.
  • B. Elastic Load Balancer/Application Load Balancer 뒤에서 m4.xlarge EC2 인스턴스에서 애플리케이션을 실행합니다. 최소 인스턴스 수 2개를 사용하여 여러 가용 영역에 걸쳐 Auto Scaling 그룹에서 EC2 인스턴스를 실행합니다. 데이터베이스를 Amazon RDS Oracle 다중 AZ DB 인스턴스로 마이그레이션합니다.
  • C. Elastic Load Balancer/Application Load Balancer 뒤의 m4.2xlarge EC2 인스턴스에서 애플리케이션을 실행합니다. 최소 인스턴스 수 1개로 여러 가용 영역에 걸쳐 Auto Scaling 그룹의 EC2 인스턴스를 실행합니다. 데이터베이스를 Amazon RDS Oracle 다중 AZ DB 인스턴스로 마이그레이션합니다.
  • D. 웹 서버 인스턴스 수를 m4.xlarge 인스턴스 2개로 늘리고 Amazon Route 53 라운드 로빈 로드 밸런싱을 사용하여 로드를 분산시킵니다. 웹 서버에서 Route 53 상태 확인을 활성화합니다. 데이터베이스를 Amazon RDS Oracle 다중 AZ DB 인스턴스로 마이그레이션합니다.

Topic #: 1

A company runs a legacy system on a single m4.2xlarge Amazon EC2 instance with Amazon EBS storage. The EC2 instance runs both the web server and a self- managed Oracle database. A snapshot is made of the EBS volume every 12 hours, and an AMI was created from the fully configured EC2 instance.
A recent event that terminated the EC2 instance led to several hours of downtime. The application was successfully launched from the AMI, but the age of the
EBS snapshot and the repair of the database resulted in the loss of 8 hours of data. The system was also down for 4 hours while the Systems Operators manually performed these processes.
What architectural changes will minimize downtime and reduce the chance of lost data?

  • A. Create an Amazon CloudWatch alarm to automatically recover the instance. Create a script that will check and repair the database upon reboot. Subscribe the Operations team to the Amazon SNS message generated by the CloudWatch alarm.
  • B. Run the application on m4.xlarge EC2 instances behind an Elastic Load Balancer/Application Load Balancer. Run the EC2 instances in an Auto Scaling group across multiple Availability Zones with a minimum instance count of two. Migrate the database to an Amazon RDS Oracle Multi-AZ DB instance.
  • C. Run the application on m4.2xlarge EC2 instances behind an Elastic Load Balancer/Application Load Balancer. Run the EC2 instances in an Auto Scaling group across multiple Availability Zones with a minimum instance count of one. Migrate the database to an Amazon RDS Oracle Multi-AZ DB instance.
  • D. Increase the web server instance count to two m4.xlarge instances and use Amazon Route 53 round-robin load balancing to spread the load. Enable Route 53 health checks on the web servers. Migrate the database to an Amazon RDS Oracle Multi-AZ DB instance.

. Amazon Route 53은 WRR(Weighted Round Robin)을 지원합니까? 예. Weighted Round Robin을 사용하면 다양한 응답이 제공되는 빈도를 지정하기 위해 리소스 레코드 세트에 가중치를 할당할 수 있습니다. 이 기능을 사용하여 A/B 테스트를 수행하고 소프트웨어를 변경한 서버로 트래픽의 작은 부분을 보낼 수 있습니다. 예를 들어 하나의 DNS 이름과 연결된 두 개의 레코드 세트가 있다고 가정합니다. 하나는 가중치 3이고 다른 하나는 가중치 1입니다. 이 경우 Route 53은 75%의 시간 동안, 25%의 시간 동안 가중치가 3인 레코드 세트를 반환합니다. Route 53은 가중치가 1인 레코드 세트를 반환합니다. 가중치는 0에서 255 사이의 숫자일 수 있습니다.

If the web client's DNS provider (or the web client OS itself) caches DNS responses and doesn't respect the TTL of your A record to properly age out the cached entry, then the client might not failover to the new IP address, because their provider could continue to reference its cached DNS entry pointing to the failed instance.

. Does Amazon Route 53 support Weighted Round Robin (WRR)? Yes. Weighted Round Robin allows you to assign weights to resource record sets in order to specify the frequency with which different responses are served. You may want to use this capability to do A/B testing, sending a small portion of traffic to a server on which you’ve made a software change. For instance, suppose you have two record sets associated with one DNS name—one with weight 3 and one with weight 1. In this case, 75% of the time Route 53 will return the record set with weight 3 and 25% of the time Route 53 will return the record set with weight 1. Weights can be any number between 0 and 255.

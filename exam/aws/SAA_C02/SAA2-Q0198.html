따라서 기본은 클라이언트 서버 모델입니다. 귀하(클라이언트)와 같은 사용자가 웹 서버에 웹 사이트를 요청할 때 임의의 높은 포트(임시 포트라고 함)를 사용하고 웹 서버는 항상 고정 포트인 443(HTTPS) 또는 80(HTTP)에 사이트를 두고 있습니다. 또는 귀하의 응용 프로그램에 따라 다릅니다. 이제 SG는 본질적으로 상태 저장형입니다. 즉, 한쪽 측면(포트 443으로 향하는 트래픽을 위해 외부 세계에서 웹 서버로 인바운드)을 허용하면 웹 서버의 응답 트래픽이 기본적으로 허용됩니다. 이는 서브넷 수준에서 적용되는 NACL과 대조됩니다. (SG는 EC2 또는 ALB와 같은 인스턴스 수준에서 적용됨) (NACL에는 봇 허용/거부 문이 있을 수 있지만 SG에는 허용 문만 있음) NACL은 상태 비저장입니다. 즉, 위의 응답 트래픽(웹 서버에서 인터넷으로)을 명시적으로 허용해야 함을 의미합니다.

gargaditya
1 year, 9 months ago
So the basics are client server model. When a user like you(client) makes request to the web server for a website, you use a random high port(called ephemeral ports) and the web server is always lsitening on a fixed port, 443(HTTPS) or 80(HTTP) or it depends on your application. Now,SG are stateful in nature meaning if I allow one side(inbound to the web server from outside world for traffic destined to port 443 ), the reply traffic from web server would be allowed by default. Contrast this to NACL ,which is applied at subnet level. (SG applied at instance level like to an EC2 or ALB) (also note NACL can have bot hallow/deny statements but SG has only allow statements). NACL is stateless,meaning we need to explicitly allow the response traffic above(from web server to the internet)

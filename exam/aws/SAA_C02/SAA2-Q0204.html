Q: RDS 인스턴스를 단일 AZ에서 다중 AZ로 변환하면 어떻게 됩니까? MySQL, MariaDB, PostgreSQL 및 Oracle 데이터베이스 엔진용 RDS의 경우 RDS 인스턴스를 단일 AZ에서 다중 AZ로 변환하도록 선택하면 다음과 같은 일이 발생합니다. 기본 인스턴스의 스냅샷이 생성됩니다. 스냅샷과 다른 가용 영역 동기 복제는 기본 인스턴스와 대기 인스턴스 간에 구성됩니다. 따라서 인스턴스가 단일 AZ에서 다중 AZ로 변환될 때 가동 중지 시간이 발생하지 않아야 합니다. 그러나 대기의 데이터가 기본 데이터와 일치하도록 따라잡는 동안 대기 시간이 증가할 수 있습니다. https://aws.amazon.com/rds/faqs/?nc1=h_ls aurora를 사용하면 다중 지역 배포에서도 매우 짧은 지연 시간을 얻을 수 있습니다.

For the RDS for MySQL, MariaDB, PostgreSQL and Oracle database engines, when you elect to convert your RDS instance from Single-AZ to Multi-AZ, the following happens: A snapshot of your primary instance is taken A new standby instance is created in a different Availability Zone, from the snapshot Synchronous replication is configured between primary and standby instances As such, there should be no downtime incurred when an instance is converted from Single-AZ to Multi-AZ. Note that read replicas can be cross az or region(Only RDS has Multi AZ for DR, not cross region for DR/Aurora has Global Aurora cross region). While both B and C will help reduced latency between end user and database replica, B also takes into account the below regarding 100ms latency for read replicas when new changes are written: ======== https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/Aurora.Replication.html The primary instance and the Aurora Replicas in the DB cluster all see the data in the cluster volume as a single logical volume. As a result, all Aurora Replicas return the same data for query results with minimal replica lag. This lag is usually much less than 100 milliseconds after the primary instance has written an update. Replica lag varies depending on the rate of database change. That is, during periods where a large amount of write operations occur for the database, you might see an increase in replica lag.

Should be "B" As an alternative to cross-Region read replicas, you can scale read operations with mimimal lagtime by using an Aurora global database. An Aurora global database has a primary Aurora DB cluster in one AWS Region and up to five secondary read-only DB clusters in different Regions. Each secondary DB cluster can include up to 16 (rather than 15) Aurora Replicas. Replication from the primary DB cluster to all secondaries is handled by the Aurora storage layer rather than by the database engine, so lagtime for replicating changes is minimal—typically, less than 1 second.

Yes, you can add up to 15 Aurora Replicas on each cross-region cluster, and they will share the same underlying storage as the cross-region replica.

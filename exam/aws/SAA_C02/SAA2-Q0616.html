주제 #: 1

회사에 고객 주문을 처리하는 애플리케이션이 있습니다. 이 회사는 주문을 Amazon Aurora 데이터베이스에 저장하는 Amazon EC2 인스턴스에서 애플리케이션을 호스팅합니다
. 때때로 트래픽이 높을 때 워크로드가 주문을 충분히 빠르게 처리하지 못하는 경우가 있습니다.
가능한 한 빨리 데이터베이스에 안정적으로 주문을 작성하려면 솔루션 설계자가 무엇을 해야 합니까?

  • A. 트래픽이 높을 때 EC2 인스턴스의 인스턴스 크기를 늘리십시오. Amazon Simple 알림 서비스(Amazon SNS)에 주문을 작성합니다. 데이터베이스 엔드포인트에서 SNS 주제를 구독하세요.
  • B. Amazon Simple Queue Service(Amazon SQS) 대기열에 주문을 작성합니다. Application Load Balancer 뒤에 있는 Auto Scaling 그룹의 EC2 인스턴스를 사용하여 SQS 대기열에서 읽고 주문을 데이터베이스로 처리합니다.
  • C. Amazon Simple 알림 서비스(Amazon SNS)에 주문을 작성합니다. 데이터베이스 엔드포인트에서 SNS 주제를 구독하세요. Application Load Balancer 뒤에 있는 Auto Scaling 그룹의 EC2 인스턴스를 사용하여 SNS 주제를 읽습니다.
  • D. EC2 인스턴스가 CPU 임계값 한도에 도달하면 Amazon Simple Queue Service(Amazon SQS) 대기열에 주문을 씁니다. Application Load Balancer 뒤의 Auto Scaling 그룹에서 EC2 인스턴스의 예약된 조정을 사용하여 SQS 대기열에서 읽고 주문을 데이터베이스로 처리합니다.

Question #: 616
Topic #: 1

A company has an application that processes customer orders. The company hosts the application on an Amazon EC2 instance that saves the orders to an
Amazon Aurora database. Occasionally when traffic is high, the workload does not process orders fast enough.
What should a solutions architect do to write the orders reliably to the database as quickly as possible?

  • A. Increase the instance size of the EC2 instance when traffic is high. Write orders to Amazon Simple Notification Service (Amazon SNS). Subscribe the database endpoint to the SNS topic.
  • B. Write orders to an Amazon Simple Queue Service (Amazon SQS) queue. Use EC2 instances in an Auto Scaling group behind an Application Load Balancer to read from the SQS queue and process orders into the database.
  • C. Write orders to Amazon Simple Notification Service (Amazon SNS). Subscribe the database endpoint to the SNS topic. Use EC2 instances in an Auto Scaling group behind an Application Load Balancer to read from the SNS topic.
  • D. Write orders to an Amazon Simple Queue Service (Amazon SQS) queue when the EC2 instance reaches CPU threshold limits. Use scheduled scaling of EC2 instances in an Auto Scaling group behind an Application Load Balancer to read from the SQS queue and process orders into the database.

선택한 답변: B
이 선택의 이유는 다음과 같습니다. Amazon SQS: SQS 대기열에 주문을 작성하면 애플리케이션이 주문 처리를 안정적이고 확장 가능한 메시징 서비스로 신속하게 오프로드할 수 있습니다. 애플리케이션은 데이터베이스 쓰기 작업이 완료될 때까지 기다리지 않고 SQS 대기열에 메시지를 보내기만 하면 트래픽을 보다 효율적으로 처리할 수 있습니다. Auto Scaling 그룹의 EC2 인스턴스: Auto Scaling 그룹을 사용하면 들어오는 워크로드에 따라 EC2 인스턴스 수를 동적으로 조정할 수 있습니다. 트래픽이 많은 기간에는 증가된 로드를 처리하기 위해 추가 인스턴스가 자동으로 시작되어 주문이 신속하게 처리될 수 있습니다. Application Load Balancer: Application Load Balancer 뒤에 EC2 인스턴스를 배치하면 여러 인스턴스에 걸쳐 트래픽을 효율적으로 배포할 수 있습니다.

Selected Answer: B
Here's the rationale behind this choice: Amazon SQS: By writing orders to an SQS queue, the application can quickly offload the processing of orders to a reliable and scalable messaging service. The application can simply send messages to the SQS queue without waiting for the database write operation to complete, allowing it to handle the traffic more efficiently. EC2 instances in an Auto Scaling group: By using an Auto Scaling group, you can dynamically scale the number of EC2 instances based on the incoming workload. During high traffic periods, additional instances can be automatically launched to handle the increased load, ensuring that orders are processed quickly. Application Load Balancer: Placing the EC2 instances behind an Application Load Balancer enables efficient distribution of traffic across multiple instances. This load balancing mechanism ensures that the processing load is distributed evenly, improving the overall performance and reliability of the system.

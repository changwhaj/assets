질문 #: 3
주제 #: 1

데이터베이스 전문가가 지난 2개월 동안 아무 문제 없이 실행되었던 여러 Aurora 복제본이 있는 Amazon Aurora DB 클러스터에서 애플리케이션 연결 실패 문제를 해결하고 있습니다. 연결 실패는 5분 동안 지속되었으며 그 후 자체적으로 수정되었습니다. 데이터베이스 전문가는 Amazon
RDS 이벤트를 검토하고 해당 시간에 발생한 장애 조치 이벤트를 확인했습니다. 장애 조치 프로세스를 완료하는 데 약 15초가 걸렸습니다.
5분간의 연결 중단의 가장 가능성 있는 원인은 무엇입니까?

  • A. 데이터베이스 충돌 후 Aurora는 마지막 데이터베이스 체크포인트에서 리두 로그를 재생해야 했습니다.
  • B. 클라이언트 측 애플리케이션이 DNS 데이터를 캐싱하고 있으며 TTL이 너무 높게 설정됨 최다 투표
  • C. 장애 조치 후 Aurora DB 클러스터는 클라이언트 연결을 수락하기 전에 워밍업할 시간이 필요합니다.
  • D. Aurora DB 클러스터에 활성 Aurora 복제본이 없습니다.

Q: Aurora는 데이터베이스 충돌 후 복구 시간을 어떻게 개선합니까? 다른 데이터베이스와 달리 Amazon Aurora는 데이터베이스 충돌 후 마지막 데이터베이스 체크포인트(일반적으로 5분)에서 다시 실행 로그를 재생하고 데이터베이스를 작업에 사용할 수 있도록 하기 전에 모든 변경 사항이 적용되었는지 확인할 필요가 없습니다. 이렇게 하면 대부분의 경우 데이터베이스 재시작 시간이 60초 미만으로 단축됩니다. Amazon Aurora는 데이터베이스 프로세스에서 버퍼 캐시를 이동하고 다시 시작할 때 즉시 사용할 수 있도록 합니다. 이렇게 하면 전압 저하를 방지하기 위해 캐시가 다시 채워질 때까지 액세스를 제한하지 않아도 됩니다.

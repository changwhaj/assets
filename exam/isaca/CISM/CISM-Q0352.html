여기서 핵심 단어는 RESTORE입니다. 백업을 복원하는 시간은 백업을 생성하는 시간과 다릅니다. 전체 백업을 생성하는 것은 증분 백업을 생성하는 것보다 시간이 더 오래 걸립니다. 그러나 전체 백업을 복원하는 것이 증분 백업을 복원하는 것보다 빠릅니다. 세부 사항은 다음과 같습니다. 전체 백업: 전체 백업은 특정 시점의 모든 데이터를 캡처합니다. 전체 백업에서 데이터를 복원하는 경우 해당 단일 백업만 복원하면 됩니다. 전체 백업을 생성하려면 더 많은 시간과 저장 공간이 필요할 수 있지만, 전체 데이터세트로 작업하기 때문에 일반적으로 복원 프로세스가 더 빠릅니다. 증분 백업: 증분 백업은 마지막 백업(전체 또는 이전 증분 백업일 수 있음) 이후 변경된 데이터만 캡처합니다. 증분 백업에서 데이터를 복원하는 경우 마지막 전체 백업을 복원한 다음 각 증분 백업을 생성된 순서대로 적용해야 합니다. 복원 프로세스는 특히 체인에 증분 백업이 많은 경우 시간이 많이 걸릴 수 있습니다. 저 역시 처음에는 혼란스러워서 증분 백업 대신 전체 백업을 선택했습니다.

The key word here is RESTORE. Time to restore a backup is different from time to create a backup. Creating a full backup takes longer than creating an incremental backup. However, restoring a full backup is quicker than restoring incremental backups Here's a breakdown: Full Backup: A full backup captures all data at a specific point in time. When restoring data from a full backup, you only need to restore that single backup. While creating a full backup may take more time and storage space, the restore process is usually faster because you're working with a complete dataset. Incremental Backup: An incremental backup only captures data that has changed since the last backup (which could be a full or a previous incremental backup). When restoring data from incremental backups, you need to restore the last full backup and then apply each incremental backup in the order they were created. The restore process can be time-consuming, especially if there are many incremental backups in the chain. I too was initially confused and i selected Full backup instead of Incremental backup.

While differential and incremental backups only back up changes made since the last backup, a full backup requires a complete copy of all data, which can be time-consuming to restore.
